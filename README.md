**RN Architecture Highlevel**
Kotlin code compiles into JS code, which for most things, is run in the JS runtime (using the Kotlin-compiled-to-JS). For areas where there are platform specific implementations, the following is true:
* Android/iOS specific implementations are exported publicly (for ease of use from RN (ie: without instantiating a full client on the native side of RN, just the JS side)) 
* The JS `actual` classes call a method on the `StytchBridge` object (JS side)
* The `StytchBridge` object is just a contract with the Kotlin/JS code that, on the JS side, there _will_ be an object that matches the declared name and shape
* The implementation of the `StytchBridge` is the output of the RN TurboModule, so updates to the `external` (kotlin) declaration and the module itself need to be kept in sync
* The `StytchBridge` methods call through the RN bridge to hand-written methods in the `android` and `ios` directories within the RN module, which in turn call the exposed implementations mentioned in bullet point 1
* The published artifacts (android/iOS) of the `stytch-multiplatform` SDK are in turn dependencies of the react-native artifact. In other words, one "flavor" of the SDK depends on two of the others
* So yeah, the way to think of it is "The SDK is Kotlin, compiled to JS, which can call a JS object generated by react-native, which can reach across the native bridge, to call platform-specific code exported by the parent SDK".
